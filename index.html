问题背景

bulk-id策略介绍

针对应用受到限制不能使用临时表的情况，hibernate在5.3.1.Final及以上版本提供了bulk-id策略，根据数据库功能定义了几个可供选择的选项：

  • InlineIdsInClauseBulkIdStrategy

  • InlineIdsSubSelectValueListBulkIdStrategy

  • InlineIdsOrClauseBulkIdStrategy

  • CteValuesListBulkIdStrategy

底层数据库必须支持可以从 non-query statements 引用的 CTE(Common Table Expressions)。例如，PostgreSQL 支持这个 feature，因为 version 9.1 和 SQL Server 自 version 2005 以来就支持它。

底层数据库还必须支持 VALUES 列表子句，如 PostgreSQL 或 SQL Server 2008。

但是，此策略需要复合标识符的 IN-clause row value 表达式，因此您只能将此策略用于 PostgreSQL。

如果你可以使用临时表，那可能是最好的选择。但是，如果不允许创建临时表，则必须选择适用于基础数据库的这四种策略之一。在考虑之前，您应该对哪一个最适合您当前的工作负载进行基准测试。例如，CTE 是 PostgreSQL 中的优化围栏，因此请确保在做出决定之前进行测量。

如果您使用的是 Oracle 或 MySQL 5.7，则可以选择InlineIdsOrClauseBulkIdStrategy或InlineIdsInClauseBulkIdStrategy。对于 MySQL 的旧版本 version，那么你只能使用InlineIdsOrClauseBulkIdStrategy。

如果您使用的是 SQL Server，则InlineIdsOrClauseBulkIdStrategy是您的唯一选择。

如果您正在使用 PostgreSQL，那么您可以选择这四种策略中的任何一种。

解决方案

根据以上介绍，考虑到项目组使用mysql,oracle这两种数据库居多，可以选择InlineIdsInClauseBulkIdStrategy策略。具体解决方案如下：

1.升级hibernate版本

<hibernate.version>5.4.8.Final</hibernate.version>

2.若项目组使用微服务（springboot）版本，在application.properties添加配置

spring.jpa.properties.hibernate.hql.bulk_id_strategy=org.hibernate.hql.spi.id.inline.InlineIdsOrClauseBulkIdStrategy

3.若项目组使用WEB（springmvc）版本，在*.spring.xml的hibernateProperties/props下添加配置

org.hibernate.hql.spi.id.inline.InlineIdsOrClauseBulkIdStrategy

​ 以上方案供有类似问题的项目组参考，在实际项目中，我们应根据使用的数据库类型和性能测试的结果决定使用哪种策略。
